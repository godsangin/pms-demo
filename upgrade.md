## 프로젝트 진척관리 방법론

### 프로젝트 관리단계 : 하위 테스크 등록 가능
 - 각 테스크 별 수행기관, 담당자, 진척률 정보 기재 가능하도록
 - 각 테스크 별로 산출물을 입고할 수 있도록 구성
 - 단위테스트, 통합테스트의 경우 결함을 등록하고 상태(조치예정, 조치 중, 조치완료)를 기입할 수 있도록 구성
1. 사업관리(점유비 10%)
 - 착수보고, 중간보고, 종료보고 진척률 입력(0 or 100%)
 - 주간보고, 월간보고 진척률 입력(프로젝트 전체 기간에 따른 진척률 자동산정)
2. 분석/설계(점유비 30%) : 정의된 현황정보를 csv로 내려받을 수 있도록 구성
 - 하위 테스크 총량 대비 진척률 평균 산정
 - 테일러링 결과에 따른 산출물 기준 진척률 입력
3. 개발/단위테스트(점유비 40%) : 정의된 현황정보를 csv로 내려받을 수 있도록 구성
 - 스프린트 기반의 주간진척 관리(각 하위 테스크 별 진척률 입력)
 - 하위테스크는 프로그램 목록으로 정의(관리화면에서는 수행기관 별 진척사항으로 표출)
 - 생산성 관리(수행기관, 담당자 별 진척률)
 - 단위테스트 관리(프로그램 목록 별 테스트 진척사항 입력, 관리화면에서 단위테스트 수행현황에 대한 UI 별도 정의)
4. 통합테스트(점유비 15%)
 - 통합테스트 시나리오를 하위 테스크로 입력
 - 각 시나리오 별 진척관리
5. 이행(점유비 5%)
 - 이행관련 하위테스트 등록, 진척관리 기능 구성


### 개선요청 프롬프트
당신은 엔터프라이즈급 B2B 솔루션을 전문으로 개발하는 '시니어 풀스택 개발자이자 소프트웨어 아키텍트'입니다. 

나는 현재 사내 프로젝트 관리를 위한 **PMS(Project Management System)**를 구축하려고 합니다. 시스템은 **3-Tier 아키텍처 (Web - WAS - DB)**로 구성되어야 하며, 아래 정의된 '프로젝트 진척관리 방법론'을 시스템에 완벽하게 구현해야 합니다. 
(기술 스택은 React, next.js, PostgreSQL을 사용합니다.)
제시된 요구사항을 분석하여 다음 4가지 결과물을 작성해 주세요.
1. **Tech Stack & Architecture**: Web(프론트엔드), WAS(백엔드), DB 기술 스택 추천 및 아키텍처 개요
2. **Database ERD / Schema**: 요구사항을 충족하는 핵심 테이블 및 관계 설계
3. **Core API Specification**: 주요 기능 처리를 위한 REST API 설계
4. **Implementation Guide**: 진척률 자동 산정, CSV 다운로드 등 핵심 비즈니스 로직 구현 방안 및 프론트엔드 UI 컴포넌트 구조
5. **프로젝트 정의 경로**: pms-demo(프론트엔드), pms-was(백엔드), DB(별도 정의 예정-postgre)

### [시스템 핵심 요구사항]

**공통 및 테스크(Task) 관리 로직**
- 모든 단계(Phase)에는 하위 테스크(Task)를 등록할 수 있어야 합니다.
- 각 테스크는 [수행기관, 담당자, 진척률, 산출물(파일/링크 입고)] 정보를 필수로 가집니다.
- 단위테스트 및 통합테스트 단계에서는 '결함(Defect)' 엔티티를 등록할 수 있어야 하며, 결함의 상태는 [조치예정, 조치 중, 조치완료] 3단계로 관리됩니다.

**프로젝트 관리 5단계 및 단계별 세부 구현사항**
프로젝트의 총 진척률은 아래 5단계의 점유비(가중치)를 곱하여 합산된 값으로 자동 계산되어야 합니다.

**1. 사업관리 (점유비 10%)**
- 착수보고, 중간보고, 종료보고는 진척률을 0% 또는 100%로만 입력할 수 있도록 유효성 검사를 추가하세요.
- 주간보고, 월간보고는 전체 프로젝트 기간(Start Date ~ End Date) 대비 현재 날짜를 계산하여 진척률이 **자동 산정(Auto-calculated)** 되도록 로직을 작성하세요.

**2. 분석/설계 (점유비 30%)**
- 등록된 현황 정보 및 진척 데이터를 CSV 형식으로 추출(Export)할 수 있는 기능을 구현하세요.
- 이 단계의 전체 진척률은 등록된 '하위 테스크 진척률의 평균'으로 자동 산정되어야 합니다.
- 시스템 테일러링 결과에 따라 필수 '산출물' 목록이 기준이 되어 진척률을 입력할 수 있도록 설계하세요.

**3. 개발/단위테스트 (점유비 40%)**
- 등록된 현황 정보를 CSV로 다운로드할 수 있어야 합니다.
- 스프린트(Sprint) 기반의 주간 진척 관리가 가능하도록 하위 테스크(프로그램 목록) 별 진척률 입력 폼을 구성하세요.
- 관리자 화면(Dashboard)에서는 '수행기관별 진척사항'과 '생산성(수행기관 및 담당자별 진척률 통계)'을 시각화할 수 있는 데이터 집계 API를 설계하세요.
- 단위테스트 관리를 위해 프로그램 목록별 테스트 진척/결함 사항을 입력하고, 단위테스트 수행 현황만 별도로 모니터링할 수 있는 독립적인 UI 컴포넌트 구조를 제안하세요.

**4. 통합테스트 (점유비 15%)**
- 통합테스트 시나리오 자체가 하위 테스크로 등록되는 구조입니다.
- 시나리오별 테스트 통과 여부 및 결함 조치 상태에 기반한 진척 관리가 이루어져야 합니다.

**5. 이행 (점유비 5%)**
- 데이터 이관, 시스템 오픈 등 이행 관련된 하위 테스크를 등록하고 진척을 관리하는 기본 기능입니다.
